TV Study Shell · Story 1.3

Safe Zone shell chrome and Surface menu

This story teaches the TV-shaped shell contract: keep critical UI inside a Safe Zone, expose one visible Focus, and let the thin DOM Shell own destination switching before Surfaces bring their own renderers.

Why this story matters

TV hardware may crop screen edges (overscan). Portfolio work that looks fine flush to a browser window can fail the “this feels like TV” interview bar. Story 1.3 makes Safe Zone and a focusable Surface menu visible in the Shell before any Surface package exists.

  • FR-3 / AD-2: Thin DOM Shell owns chrome and navigation chrome.
  • FR-1: Entry screen lists Home, Live, EPG, and WebGL Lab.
  • Interview value: you can explain overscan, focus ownership, and why Shell ≠ Surface runtime.

Overscan vs Safe Zone

Overscan is manufacturer-dependent cropping around the displayable area. Safe Zone is the inner region where critical UI should live so text and focus rings are not clipped.

  • Common convention: ~5% margin per edge (inner ~90%).
  • At 1920×1080 that is roughly 48px horizontal and 27px vertical.
  • Backgrounds may bleed to the edge; focused items and labels should not.

In this repo the Safe Zone is intentionally visible (dashed guide + label) so the teaching overlay is obvious on a desktop mock. Invisible padding alone would not prove the concept.

Thin DOM Shell vs Surface renderers

The Shell is Vite + TypeScript DOM. Surfaces later bring Canvas, raw WebGL, Blits, or Solid. The Shell must not become a React/Solid app “for convenience.” 

menu (Shell) → select → host.mount(surface)   // Story 1.4+
surface root Back → menu                      // AD-4 loop later

Story 1.3 stops at menu + acknowledge. Select ≠ mount. Saying “I selected WebGL Lab” in the status line is honest; inventing a fake mount API here would steal Story 1.4.

Single visible Focus + shared D-pad

TV UIs have one primary focused element. The menu consumes getDpadAction from @tvshell/shared so Arrow keys / Enter / Back stay aligned with later Surfaces and Playwright. 

const action = getDpadAction(event);
// up/down → move focus index
// select → acknowledge SurfaceId
// back → stay on menu (until Surfaces exist)

Prefer an explicit focus index + high-contrast .is-focused style (10-foot readability) rather than hoping default browser focus looks TV-ready.

Honesty bounds

  • Desktop Chromium + keyboard is a proxy, not a certified Tizen/webOS remote.
  • 5% Safe Zone is a portfolio convention, not a measured guarantee on every TV.
  • Four destinations are labeled now; packages for those Surfaces arrive in later epics.
  • Mount/unmount cleanup and Back-from-Surface are intentionally deferred to Story 1.4+.

Whiteboard talking points

  1. What is Safe Zone? Keep critical UI inside ~5% insets so overscan does not crop focus and text.
  2. Why visible guide? On a desktop mock, a teaching overlay proves the inset exists; padding alone is easy to miss.
  3. Why thin DOM Shell? Navigation chrome should not import Canvas/WebGL/Blits/Solid runtimes.
  4. What is deferred? Real Surface mount/unmount, OEM remote codes, and visual polish beyond Safe Zone.
