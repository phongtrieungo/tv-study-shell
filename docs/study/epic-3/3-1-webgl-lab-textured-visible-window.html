WebGL Lab · Textured Visible Window

Canvas EPG already taught virtualization. Interviewers still ask for GPU pipeline vocabulary — buffers, textures, shaders, draw calls — and what Lightning/Blits abstracts. Lab W closes that gap with the same Visible Window math on raw WebGL. See docs/webgl-investment.md.

Why Lab W exists

  • Canvas alone proves you can window a big guide — it does not prove you can talk about the GPU.
  • Blits/Lightning is production-shaped WebGL, but the scene graph can hide the metal until FPS dies and you cannot debug.
  • Raw WebGL Lab lets you say: “I virtualized an EPG in Canvas, then rendered the same Visible Window idea with WebGL textures and draw calls.”

How “navigation” works without HTML buttons

Interview line: App-owned focus. The remote never focuses a DOM node inside the grid. We keep two numbers in JS (focusChannelIndex, focusTimeMs), move them on D-pad, then redraw so the matching program quad is brighter.

  1. No hit-testing DOM. Program cells are triangles in a VBO. There is nothing for the browser to tab to.
  2. Input path: keydown (capture) → getDpadAction → handleAction updates focus → schedulePaint.
  3. Back is Shell-owned: Lab ignores back so it bubbles to Shell host.leave() → unmount → disposeGpu.
  4. Visual focus: In drawVisibleWindow, if the program contains the focus time on the focus channel, that quad’s a_focus is 1; the fragment shader boosts RGB. A thin focus-time cursor quad is also drawn.
  5. Window follows focus: Changing focus re-runs computeVisibleWindow (anchor center), so the Visible Window scrolls with you — still only drawing the intersecting subset.

Unmount (cleanup closes the loop)

Back / Shell leave
  → index.ts unmount()
  → remove keydown + ResizeObserver + cancel RAF
  → gl.ts disposeGpu()   ← deleteBuffer / deleteTexture / deleteProgram / deleteShader
  → hasActiveSideEffects() === false

Same math, different renderer

  • EPG uses CanvasRenderingContext2D (CPU path, easy text/labels).
  • Lab uses WebGLRenderingContext / WebGL2RenderingContext (GPU path).
  • Both import computeVisibleWindow from @tvshell/shared — windowing is a data problem; Canvas vs WebGL is a renderer choice.
  • Story 3.2 will measure both side-by-side; Story 3.3 deepens the vocabulary write-up. This page is the pipeline practice story.

Whiteboard talking points

  • Why not DOM cells? Same as EPG — thousands of nodes hurt TV Chromium. WebGL does not magically virtualize; you still must draw only the Visible Window.
  • Why delete* on leave? Dropping JS references does not free GPU memory. Surface leave must call deleteBuffer / deleteTexture / deleteProgram (AD-6 / AD-9).
  • What Lightning abstracts: scene graph + WebGL (or Canvas fallback). Lab W is the metal; Blits Home (Epic 4) is the applied scene graph — not a replacement for this package.
  • Input ownership: Shell Back → leave; Lab owns ↑↓←→ while mounted (capture listener), same contract as EPG.

Demo script (desktop proxy)

  1. Open Shell → focus WebGL Lab → Enter.
  2. Confirm HUD: drawn ≪ logical 600 and context webgl2 (or webgl).
  3. Hold ↑↓ / ←→ — window ranges change; textured tiles move; you never draw all 600.
  4. Back — unmount; console cleanup probe should report no active side effects.
  5. Remount; then check EPG still works (registry-swap only touched webgl-lab).

Honesty bounds

  • This is a learning lab (textured 2D UI tiles) — not a claim of graphics-engine expertise.
  • Desktop Chromium + keyboard D-pad ≠ OEM TV memory / GPU driver truth.
  • No Three.js here on purpose — the JD probe is pipeline literacy, not “I used a 3D framework.”
  • Context-loss recovery is awareness-level only; full rebuild frameworks are out of scope.
