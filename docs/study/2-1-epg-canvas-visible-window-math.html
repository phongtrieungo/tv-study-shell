TV Study Shell · Story 2.1

EPG Canvas · Visible Window math

Interviewers ask how you keep a 50×24h guide responsive. The answer is not “Canvas is fast” — it is Visible Window virtualization: compute which channels and time range are on screen, draw only that subset, and prove it with a draw count.

Mental model

  1. Logical grid — every channel × every program in fixtures (here: 50 channels × 12 two-hour blocks = 600 program cells).
  2. Visible Window — the channel index range and time range currently in view (driven by focus / scroll offsets + viewport size).
  3. Draw set — programs that intersect the window. HUD / log must show drawn ≪ logical (e.g. tens vs 600).

Whiteboard talking points

  • Why not DOM cells? Instantiating thousands of nodes for an EPG blows memory and layout cost on TV-class Chromium. Canvas (or scene-graph) + windowing is the classic pattern.
  • Why shared math? WebGL Lab (Story 3.1) should reuse the same computeVisibleWindow — GPU vs CPU is a renderer choice; the window is a data problem (AD-9).
  • What updates the window? Arrow focus moves channel / time → recompute → redraw. Full program-detail UX is a later story; the teaching point here is the window, not a polished EPG product.
  • Cleanup ownership: Shell Back → unmount; EPG removes its key listener and cancels RAF (AD-6). Shell keeps Back; Surface owns arrows while mounted.

Demo script (desktop proxy)

  1. Open Shell → focus EPG → Enter.
  2. Read the HUD: drawn cell count vs logical 600.
  3. Hold ↑↓ / ←→ — window ranges and draw count change; you never draw the full 600.
  4. Back — Surface unmounts; menu returns. Remount to prove lifecycle.
  5. Honesty line: this is desktop Chromium + keyboard D-pad, not OEM TV memory truth.

API shape to remember

  • computeVisibleWindow({ focusChannelIndex, focusTimeMs, viewportChannelCount, viewportDurationMs, totalChannels, dayStartMs, dayEndMs, anchor })
  • Returns { channelStart, channelEnd, timeStartMs, timeEndMs } — clamped, no negative indices, no overrun past the schedule.
  • programsInVisibleWindow / countProgramsInVisibleWindow for draw lists and HUD accounting.

Honesty bounds

  • Desktop Chromium + keyboard ≠ Tizen/webOS remote or device heap.
  • HUD draw counts are teaching evidence, not a labeled Perf Note yet (Story 2.4).
  • Vitest for this math lands in Story 7.1 — keep functions pure so CI can own them later.
