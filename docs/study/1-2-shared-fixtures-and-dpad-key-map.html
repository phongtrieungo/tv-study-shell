TV Study Shell · Story 1.2

Shared fixtures and D-pad key map

This story teaches the interview-important split between app vocabulary and surface behavior: every surface should consume the same channels, programs, rails, and D-pad semantics before UI-specific focus logic exists.

Why this story matters

Smart TV surfaces break down quickly if each surface invents its own data shape or key handling. Story 1.2 centralizes those contracts inside @tvshell/shared so the shell, EPG, Home rail, and future tests all speak the same language.

  • AD-3: Mock Data is the only data plane in v1.
  • AD-4: D-pad key mapping is shared and stable across surfaces.
  • Interview value: you can explain how shared contracts reduce drift before complex rendering work starts.

Shared fixtures mental model

The fixtures are not random demo blobs. They model the minimum data integrity that later stories need:

  • channels provide stable row identities for the EPG.
  • programs cover a full 24-hour schedule with deterministic ISO timestamps.
  • homeRails provide the shared poster vocabulary for Blits Home later.

In interviews, the key point is not “I had 50 channels.” The key point is “I designed fixtures that make virtualization math and focus behavior possible without a backend.” 

type Channel = { channelId: string; name: string; number: number };
type Program = {
  programId: string;
  channelId: string;
  title: string;
  start: string; // ISO-8601
  end: string;   // ISO-8601
};
type Rail = { railId: string; title: string; items: RailItem[] };

D-pad mapping: use KeyboardEvent.key, not keyCode

Desktop Chromium is the deliberate TV proxy in this repo. That means the shared map should be built around current web-standard key strings:

  • ArrowUp, ArrowDown, ArrowLeft, ArrowRight
  • Enter for Select
  • Backspace and Escape for Back

The important design choice is the semantic layer: surfaces should depend on up, down, left, right, select, and back, not on raw key strings sprinkled everywhere. 

function normalizeDpadKey(key: string): DpadAction | null {
  switch (key) {
    case 'ArrowUp': return 'up';
    case 'ArrowDown': return 'down';
    case 'ArrowLeft': return 'left';
    case 'ArrowRight': return 'right';
    case 'Enter': return 'select';
    case 'Backspace':
    case 'Escape': return 'back';
    default: return null;
  }
}

Determinism and honesty bounds

  • Use a fixed schedule anchor instead of Date.now() so smoke checks are repeatable.
  • Use slug IDs such as channel-01 and ISO timestamps so later stories can reason about stable references.
  • This is a desktop-keyboard proxy, not a claim that OEM remote edge cases are fully solved.
  • The story intentionally does not implement Safe Zone, focus movement, or menu behavior yet.

Whiteboard talking points

  1. Why centralize fixtures? Because the EPG, Home, tests, and shell all need the same domain vocabulary before renderer-specific work begins.
  2. Why centralize input? Because TV bugs often come from inconsistent Back handling and divergent key semantics across surfaces.
  3. Why deterministic data? Because virtualization math and tests are easier to trust when the dataset is stable.
  4. What is intentionally deferred? Safe Zone UI, focus graphs, Visible Window math, Playwright smoke, and OEM-specific remote integration.
