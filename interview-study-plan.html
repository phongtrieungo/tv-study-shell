4 Browser rendering pipeline (must-know)

Interviewers expect you to explain why TV apps avoid heavy DOM and prefer canvas/WebGL.

User input → JavaScript → Style → Layout → Paint → Composite ↑ ↑ reflow costly layer / GPU compositing DOM-heavy SPA pain on TV: • Many nodes → layout + style recalc • Large images → memory pressure • Animations on layout props → jank Canvas / WebGL path: • Fewer DOM nodes; draw commands on GPU • Better control over frame budget (16ms @ 60fps, 33ms @ 30fps TV)

Core concepts

  • Critical rendering path: HTML → CSSOM → render tree → layout → paint
  • Reflow vs repaint: geometry change vs visual-only change
  • Compositor layers: transform/opacity cheaper than top/left/width
  • Main thread vs GPU: what blocks scrolling/animation
  • requestAnimationFrame: sync draws to display refresh

Connect to your Angular work

  • OnPush change detection = fewer unnecessary renders
  • trackBy = stable list identity, less DOM churn
  • Virtual scroll = EPG-like long lists without full DOM
  • Lazy routes / micro-frontends = memory isolation on TV
  • SignalR partial updates = Live TV data refresh pattern

5 Performance engineering toolkit

Profiling workflow (memorize this loop)

  1. Measure — Lighthouse, Performance panel, Memory heap snapshots
  2. Identify — long tasks, layout thrashing, GC spikes, image leaks
  3. Hypothesize — DOM count? rerenders? uncached textures?
  4. Fix — virtualize, batch, pool objects, reduce layers
  5. Verify — compare filmstrip / FPS / heap after fix

Memory on constrained devices

  • Object pooling for list tiles / EPG rows
  • Dispose textures, event listeners, subscriptions
  • Avoid retaining detached DOM nodes (heap snapshot → Detached)
  • Image sizing: serve TV-resolution assets, not 4K web assets
  • Code-splitting; unload off-screen routes

Responsiveness patterns

  • Defer non-critical work (requestIdleCallback where supported)
  • Web Workers for heavy parsing (EPG JSON)
  • Incremental render: paint visible window first
  • Input latency: D-pad focus must feel instant on TV
Bring numbers to interviews Even rough metrics help: “Reduced initial render nodes by virtualizing grid”, “Cut bundle chunk by lazy-loading map module”, “Eliminated memory leak from unsubscribed observables.”

6 WebGL & Canvas fundamentals

Nice-to-have on paper — often essential in practice for this role.

Canvas 2D (stepping stone)

  • getContext('2d') — immediate mode drawing
  • Good for UI sprites, text, simple grids
  • CPU-bound at scale; know limits
  • Double buffering / offscreen canvas patterns

WebGL basics

  • GPU pipeline: vertices → shaders → rasterization → framebuffer
  • Buffers, textures, draw calls, blend modes
  • Batch draws to reduce call overhead
  • Power-of-two textures; mipmaps; atlases
  • WebGL1 vs WebGL2 on TV browsers (check platform)

Mini mental model: DOM vs Canvas vs WebGL

DOM/React/Angular     → great for forms, accessibility, rapid product UI
Canvas 2D             → custom grids, charts, lightweight game-like UI
WebGL                 → heavy animation, video surfaces, large visual catalogs
Lightning / Blits     → TV-optimized scene graph + GPU rendering abstraction
Week 3 lab Build a virtualized EPG grid in Canvas 2D: 7 days × 48 half-hour slots, only render visible rows/columns, navigate with arrow keys. Then explain how you'd move it to WebGL/Lightning.

7 SolidJS, Lightning V3, Blits

JD names these explicitly — understand philosophy, not just syntax.

Tech What it is Why TV teams use it Study action
SolidJS Fine-grained reactive UI (signals), no virtual DOM Predictable updates, less render overhead than React Build counter + list app; compare to Angular change detection
Lightning V3 WebGL/canvas TV UI framework (successor to LightningJS) Built for STB/TV performance, focus, shaders, textures Clone starter; render row of focusable tiles with D-pad
Blits Lightweight TV app framework (Philips / Metrological ecosystem) Component model optimized for embedded TV browsers Read docs; compare component lifecycle to Lightning

SolidJS — key ideas

  • createSignal, createEffect, createMemo
  • UI updates trace to precise dependencies
  • No VDOM diff — important talking point vs React
  • JSX feels React-like but execution model differs

Lightning V3 — key ideas

  • Scene graph: parent/child elements, transforms, z-index
  • Focus management for remote control navigation
  • Textures from images; smooth focus/scroll animations
  • Often paired with app routers and data layers
Interview bridge from Angular “Angular taught me to control change propagation and bundle size. SolidJS/Lightning push that further by avoiding VDOM or DOM altogether — that's the direction TV apps need.”

8 Smart TV platform constraints

Hardware reality

  • Low RAM (often 512MB–1.5GB for app)
  • Weak CPU; GPU varies by chipset
  • 30fps or 60fps targets; dropped frames very visible at 3m viewing distance
  • Cold start time matters (app launch from launcher)

Platform landscape

  • Tizen (Samsung) — web apps, specific APIs
  • webOS (LG)
  • Android TV / Fire TV — WebView or native hybrid
  • Embedded Chromium versions often < desktop evergreen
  • Certification & memory limits per store

TV UX vs web UX

  • D-pad navigation: focus ring, spatial navigation, no hover
  • Safe zones: overscan margins on older TVs
  • 10-foot UI: larger type, fewer elements per screen
  • Back button behavior: predictable stack navigation
  • Long session: memory leaks become showstoppers

11 Likely interview questions

Why canvas/WebGL instead of React for TV?
DOM/React overhead, memory from many nodes, layout cost, unpredictable rerenders. Canvas/WebGL gives controlled draw budget and better GPU use on weak hardware.
How would you optimize an EPG grid?
Virtualize rows/columns, pool cells, batch draws, prefetch data windows, separate static chrome from animated “now” indicator, profile memory over 30+ min session.
Explain a performance bug you fixed.
Use Switzerland or Australia story; structure: symptom → profile → root cause → fix → verification.
Experience with WebGL / Lightning / Smart TV?
Honest: production strength in TS performance UI; active hands-on with Lightning/SolidJS/canvas labs; strong transfer from map/grid/live-update work.
How do you approach D-pad focus navigation?
Spatial focus graph, visible focus states, predictable wrap/escape, restore focus on back navigation, minimize focus traps, test on device not just desktop keyboard.
SolidJS vs React — why does it matter here?
Fine-grained reactivity avoids VDOM diff cost; updates only what changed — aligns with TV frame budget thinking.
